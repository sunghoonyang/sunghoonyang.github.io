DAG generator (DAGEN)

  1. Jinja
  2. Airflow Custom Operator (S3 to Hive, S3 to Redshift, Redshift to Redshift)
  3. Hive & Redshift as target clusters
  4. Jenkins
  5. AWS SQS
  6. S3

This project was conceived to automate the tedious process of creating DAGs for lift-and-shift jobs. Lift and shift jobs are tedious in a sense that they are merely copying the incremental data from the source to the target’s staging table, and merging the staging table with the target table. Lift-and-shift jobs can be administered in a meta-driven way, and meta-driven way proves indispensable to serve stakeholders +100 source Aurora Databases and +2000 tables: and do this with negligible delay. The tables will each have in meta database parameters for ETL, such as SQL statement for the data extraction with parameterized start and end timestamp, basis timestamp column, source database and schema, and target table & schema name should they be different from the source. Finally, each job will also have the jinja template category. This allows us to use different Jinja templates for each type of lift-and-shift.

Once the meta database is live, Dagen uses SqlAlchemy’s declarative_base to bring to the Flask session the meta tables as ORM. The Relationship is set up such that each DAG object will have child tasks, which in turn has child task parameters. Dagen uses ORM data to automatically fill out the Jinjafied DAGs. the templates are filled out according to the job’s template category. For example, if the category is rs_to_rs (Redshift to Redshift), its task will have disparate tasks, such as the in-house developed BetweenRedshiftOperator that encapsulates different loading types ( type 1 : snapshot - truncate insert. type 2 : master table - delete target by staging table’s primary key and insert, type 3 : fact table - update if staging.modified_at_timestamp > target.modified_at_timestamp for each primary key). From the import statements to each Airflow Operator’s parameters, all necessary variables are read from meta database and written to python file.

Once the file generation is finished, it is uploaded to a specific S3 bucket, and invokes SQS message. This calls the standard Jenkins procedure that reads from S3 bucket and automatically merge with Airflow-DAGs git project if Jenkins testes pass.

This project successfully automated all Aurora to Redshift, Redshift to Redshift, Aurora to Hive and Redshift to Hive lift-and-shift DAG generation, and generated 4000 DAGs in less than a year.

Meta-driven IBM DataStage Job Runner

  1. IBM DataStage
  2. Python3.6 with MySQLdb connection, boto3
  3. Flask Web App

mySQL table has all the information of DataStage tasks (source db, query to be sent, start/end timestamp, primary key etc.) Python script in DataStage server dynamically creates a parameter set, and runs on cli DataStage jobs (using dsjob) in their respective frequency & ETL group. The csv file created will be uploaded to designated S3 location.

This project was done to reduce latency between individual job’s finish time and group-wise AWS S3 upload. By dynamically running DataStage jobs as above, each finished DataStage task is immediately followed by S3 upload. The meta-driven ETL approach also allows for easy creation of a new job. I took advantage of this to distribute ETL job creation task to all DW engineers by providing them with Flask-powered job creation form, where they can fill out the form that collects all the meta related parameters, and generate all necessary tasks for them, including corresponding Airflow task and Redshift table.

Acquisition DB Version 3

  1. Airflow
  2. Redshift, S3, EC2

Airflow has been adopted as the teamwide scheduler, and the ETL tasks were migrated to Airflow. This means that the SQL procedures were saved as files, called using PostgreSQL hooks in Airflow Operators, and downstream tasks, such as S3 flag creation, downstream job starts, are followed. The migration to Airflow also gave rise to the Dagen project. (DAG + generation, where DAG is a unit of ETL task) Using Airflow Jinja templating, the dag definition files are dynamically created using Meta table. The meta table would have dag specific information (task owner, source database, target database, upstream task, type of load, frequency of load, etc.) and create dag file accordingly. Dynamically created files are loaded to S3, where it is tried for Jenkins build test. If the dags pass Jenkins test, they are added to Airflow server’s dagbag directory location, and synced to the scheduler.

Acquisition DB Version 2

  1. SQL-SERVER, bcp
  2. NZSQL (EXTERNAL TABLE via NZ_UNLOAD)
  3. python packages (e.g. psycopg, sqlalchemy, boto)

The below project synopsis was written when the Acquisition DB was still at the beta version. The initial effort was focused on Redshift, whose scheduled tasks run by crontab jobs in an EC2 instance.

Having Redshift cluster as a centerpiece of ODS, another requirement was given, that the specific transactional records be easily queried. In order to satisfy this, previously deprioritized developing SQL SERVER acquisition DB was resumed. CTDS was first tested, but the throughput was nowhere near that of BCP. A few engineers worked together to resolve the issue with BCP - the order of the column is not respected by the BCP ingestion and the load fails when the files are big. We circumvented the problem by splitting files in 32MB bite-size. This allowed me to take advantage of the relatively fast BCP utility (although the speed is nowhere near platform native Redshift).

Acquisition DB Version 1

  1. Redshift, S3, EC2, boto3, AWS CLI
  2. sqoop
  3. SQL-SERVER, bcp
  4. pyodbc, pymssql, ctds, sqlalchemy

Acquisition DB is Coupang’s effort to separate the Operational data Storage (ODS) from mart data. The on-premise netezza PDA has been overloaded due to fast growth of the company, which was accompanied by inability to create sandbox-ed data sources for different teams with analysis needs. netezza PDA handled all analyses from catalog team to purchase order, thereby demanded to house all ODS. Acquisition DB is built to relieve netezza of such heavy-lifting, and separate data source for different domains. As each domain would have needs for different ODS tables, multiple instances of Redshift will be ODS for different domains. We considered two candidates: sqlserver and redshift. Sql Server was built mainly with ctds library which supports pseudo-bulk insert (where python iterator object of rows are batch processed and performed well (50k rows per sec -39 columns) However, due company’s direction of its architecture to migrate completely to AWS, redshift was eventually selected. In order to build ODS on Redshift, its COPY command from S3 was leveraged. Sqoop, from two servers that are on PDA ACL would import tables (which are audited with respect to max pk & row count), format individual files with GNU utilities (sed & awk) and uploaded to designated S3 path using AWS CLI. This module was built by with python, using boto3 and subprocess to call on GNU utilities. Once its on S3, it is copied to Redshift via crontab schedule from an EC2.

Web App with OracleDB for DW ETL Stakeholders

  1. Nodejs with Express
  2. OracleDB
  3. CentOS 6.4 Server

I developed a nodeJS web app that allows ETL job stakeholders to view information about jobs in a paginated table, and choose jobs that they are interested in to request SLA alert (Talend job checks the SLA fact-table that has mapping between stakeholder and followed jobs. stakeholders receive personalized email alert should the ETL job fail to meet each stakeholder’s given SLA time) at chosen time. The application asynchronously queries data from OracleDB, and populate data in table format as well as graphically, using HighCharts. The spline/column chart will show job duration in seconds, as well as start/end time of the respective job, in 7 days recent trend.

The web app is structured such that the users, upon landing in ./login path, must go through Company LDAP. Session middleware checks the user status and current url to protect the app from unauthorized access. If an LDAP authenticated user is not enrolled in meta table for the user accounts, session middleware redirects to the sign up form, and user can request membership. Admins, on the other hand, have admin page open to access member-request table to grant specific users application permission.

Loading & Running the node.js app that uses oracledb module on CentOS 6.4 is tricky because oracledb module has dependency to gcc version 11. Since upgrading the server OS to CentOS 7 was not a possible choice, this problem was circumvented by using devtoolset-3, an unofficial package that updates the session’s c++ compiler. I first had created a bash script that sets LD_LIBRARY_PATH, enables devtoolset-3 ( scl enable devtoolset-3 bash )  and runs node.js app via npm start as a daemon. But this is less than ideal because it does not guarantee app restart should stakeholder activity generate error and app terminates. This was dealt with by using pm2 and setting up environment variable and package.json, and devtoolset-3 enabling with shelljs package in express.config.js file.

DB Monitoring with Icinga2

  1. Docker
  2. cx_Oracle
  3. OracleDB
  4. Icinga2Web

Icinga is an app that makes DB monitoring easy. A user is responsible for defining a few objects, and the rest is done by Icinga. User is responsible for writing the check that have logic inside that determines if the job is complete for particular service. The check is to return values from 0 to 3, each meaning Complete, Warning, Critical, Unknown respectively. User also creates service object that contains parameters for the check for the particular service. For example, a ETL migration of Sales mart table may be parameters such as jobid, user type, and so on, which are passed into the check script. Check script that runs custom logic. For example, we could use Nagios to pass in threshold rowcount as a minimum requirement for the job to pass (sales mart row count has to be > 100000 per day).

Icinga is mounted on a docker image, as well as oracle instantclient, mySql as the server side db and and Ubuntu(Debian) for ssh. The scripts are written in Python, and connects to OracleDB via cx_Oracle. In my scope of project, we made use of the existing job meta schema in OracleDB, and analyzed a single table row by row to as a single row equates to single job or service.

OracleDB schema backup

  1. AWS OracleRDS
  2. S3 Bucket
  3. Talend

A project to have a backup for ETL meta, housed in OracleDB. Create an OracleRDS instance, and set it up so that it is a replica of the on-premise ETL meta schema (not just DDL but copying functions, objects, SP and so on.) create a snapshot of it, and delete the instance.

Create a Talend job that copies the meta tables (dimension table for jobs, log tables for running jobs, data quality dimension tables, etc) and periodically save the data as a file to S3 bucket. Also create a talend job that inserts these files to respective tables in OracleRDS

When OracleDB is down, have the snapshot rebooted, and the files in S3 are inserted into respective tables in OracleRDS. Change the server parameter of OracleDB to point to DNS name of OracleRDS.
